# Railway-hosted OAuth broker plan

## Decision

Use a small Railway-hosted OAuth broker as the preferred convenience path for Mattergate connector installs.

BYO OAuth credentials remains the fallback in an **Advanced setup** path, but the normal user-facing flow should be:

```text
Self-hosted Mattergate → Railway OAuth broker → vendor consent → Railway OAuth broker → self-hosted Mattergate
```

The broker exists to keep maintainer-owned vendor OAuth client secrets out of open-source deployments while still letting admins click **Integrate Lawmatics** instead of creating/pasting developer credentials.

## Why Railway is acceptable

Dan already has a Railway account running multiple apps, so the usual $5/month floor is not a deciding factor. The broker should be very cheap because it is not in the hot path for MCP traffic. It only handles install/reconnect/revoke-style flows.

Expected usage is tiny:

- one or a few OAuth installs per firm
- occasional reconnects
- lightweight state and handoff records
- no normal Lawmatics API proxying
- no long-running background worker unless later needed

Railway is also operationally consistent with Dan's normal GitHub → Railway auto-deploy workflow.

## Repository and project shape

Use a separate repo/service for the broker, likely:

```text
dschnurbusch/mattergate-oauth-broker
```

Keep it separate from the main `mattergate` repo because the broker is a different trust boundary:

- It owns maintainer/vendor OAuth client secrets.
- It has a different deploy cadence from the gateway.
- It should be extremely small and easy to audit.
- It should not be included in the self-hosted Railway template.
- It can later support a hosted/cloud Mattergate product without contaminating the self-hosted gateway boundary.

The broker code can still be public/open-source if desired. Secrets live only in Railway variables.

## Railway deployment shape

Initial Railway resources:

1. **Broker web service**
   - Deployed from GitHub `main` using Railway's normal source integration.
   - Exposes only OAuth broker routes and health checks.
   - Stateless except for short-lived install state/handoff storage.
2. **Postgres or Redis-compatible store**
   - Use Railway Postgres first unless a lighter built-in durable store emerges.
   - Tables hold short-lived OAuth state, installation session metadata, redirect allowlist entries, and audit metadata.
   - No long-term Lawmatics access token storage by default.

Avoid a worker process for v1.

## Required broker environment variables

| Variable | Purpose |
| --- | --- |
| `DATABASE_URL` | Short-lived state, handoff, allowlist, audit metadata. |
| `APP_URL` | Canonical broker URL, e.g. `https://oauth.mattergate.dev`. |
| `SESSION_SIGNING_SECRET` | Signs broker state/session cookies if any. |
| `STATE_SIGNING_SECRET` | Signs OAuth state and prevents tampering/replay. |
| `HANDOFF_SIGNING_SECRET` | Signs one-time handoff codes/tokens. |
| `LAWMATICS_CLIENT_ID` | Maintainer-owned Lawmatics OAuth app client ID. |
| `LAWMATICS_CLIENT_SECRET` | Maintainer-owned Lawmatics OAuth app client secret. |
| `LAWMATICS_CALLBACK_URL` | Broker callback registered with Lawmatics, derived from `APP_URL`. |
| `ALLOWED_INSTANCE_DOMAINS` | Optional coarse allowlist for self-hosted Mattergate callback origins. |
| `LOG_LEVEL` | Default `info`; never log secrets or tokens. |

## Flow

### 1. Instance registration / install start

Self-hosted Mattergate starts an install session:

```http
POST https://oauth.mattergate.dev/api/install/start
```

Payload should include:

- instance URL
- requested connector (`lawmatics`)
- instance public key for encrypted handoff, if implemented
- nonce/challenge generated by the Mattergate instance
- admin return URL

Broker validates:

- URL is HTTPS outside local development
- callback host is allowed or passes a registration challenge
- connector is supported
- nonce/state is fresh

### 2. Vendor authorization

Broker redirects admin to Lawmatics:

```text
https://app.lawmatics.com/oauth/authorize?client_id=<BROKER_CLIENT_ID>&redirect_uri=<BROKER_CALLBACK>&response_type=code&state=<SIGNED_STATE>
```

Lawmatics only needs one callback URL:

```text
https://oauth.mattergate.dev/api/oauth/lawmatics/callback
```

### 3. Token exchange

Broker validates signed state, then exchanges the code server-side:

```http
POST https://api.lawmatics.com/oauth/token
Content-Type: application/x-www-form-urlencoded

client_id=<BROKER_CLIENT_ID>
client_secret=<BROKER_CLIENT_SECRET>
grant_type=authorization_code
code=<AUTH_CODE>
redirect_uri=<BROKER_CALLBACK>
```

### 4. Handoff to self-hosted Mattergate

Broker creates a short-lived, one-time handoff record.

Preferred shape:

- encrypt Lawmatics token material to the Mattergate instance public key
- store encrypted blob for 5 minutes
- issue a one-time handoff code to the admin browser
- browser redirects back to the Mattergate instance with only the handoff code
- Mattergate redeems the handoff code server-to-server
- broker deletes the record immediately after successful redemption

Fallback shape, if public-key handoff is not ready:

- handoff code redeems once over TLS
- broker returns token material directly to the Mattergate server
- record is deleted immediately

Do not put the Lawmatics access token in a browser-visible URL fragment or query parameter.

## Security rules

- Broker does not proxy normal MCP traffic.
- Broker does not proxy normal Lawmatics API calls.
- Broker does not store vendor access tokens long-term for self-hosted installs.
- Broker never returns or logs vendor client secrets.
- OAuth codes, handoff codes, and state records expire quickly.
- Handoff codes are one-time use.
- Redirect destinations are validated against registered instance URLs.
- Audit only metadata: connector, instance ID/domain, timestamps, result, failure reason class.
- No matter/client/vendor record data belongs in broker logs.
- Rate-limit install starts, callbacks, and handoff redemption.
- Add a kill switch for a compromised connector/client secret.

## Advanced setup fallback: BYO OAuth credentials

Self-hosters who do not want to trust the broker, or who need their own vendor app for compliance/review reasons, can use BYO credentials.

Advanced flow:

1. Self-hoster creates or obtains a Lawmatics OAuth app.
2. Self-hoster configures `LAWMATICS_CLIENT_ID` and `LAWMATICS_CLIENT_SECRET` in their Mattergate deployment.
3. Mattergate uses its own callback URL directly:

   ```text
   ${APP_URL}/api/connectors/lawmatics/oauth/callback
   ```

4. The broker is bypassed entirely.

This should live in the UI under **Advanced setup** and in docs under a BYO credentials section.

## Future hosted/cloud Mattergate

A future managed Mattergate service can reuse the same broker trust model or absorb it into the hosted app. Keep the broker separate for now so self-hosted Mattergate, the broker, and eventual hosted Mattergate can evolve without blurring boundaries.

Potential future services:

```text
mattergate              # open-source self-hosted gateway
mattergate-oauth-broker # maintainer OAuth broker for connector install convenience
mattergate-cloud        # optional hosted managed service later
```
